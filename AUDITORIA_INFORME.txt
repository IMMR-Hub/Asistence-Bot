1. RESUMEN EJECUTIVO
================================================================================

Se ha completado exitosamente la Fase 1 del proyecto Universal Sales Automation
Core: un sistema de automatización de ventas multi-tenant, modular y config-driven,
basado en arquitectura hexagonal.

El sistema procesa mensajes de WhatsApp y Email, clasifica intenciones con IA
(LLM local), genera respuestas contextualizadas, gestiona escalados a humanos y
registra toda la actividad en una base de datos relacional.

PRINCIPIO FUNDAMENTAL: PLUG AND PLAY
El core del sistema NUNCA se modifica. Nuevos clientes se añaden únicamente
mediante configuración JSON en la base de datos. Un script de Setup (PowerShell)
con apenas 3 pasos permite onboarding de nuevos clientes sin tocar el código.

2. ARQUITECTURA IMPLEMENTADA
================================================================================

Arquitectura Hexagonal (Ports & Adapters) con separación clara entre capas:

CAPAS:
- Adapters: Webhooks, WhatsApp (Meta), Email (SMTP)
- Ports: Interfaces de servicios y repositorios
- Services: Orquestación (MessageProcessor, TenantService)
- Modules: Lógica de negocio (8 módulos independientes)
- Database: SQLAlchemy + PostgreSQL (10 modelos)
- Config: Settings centralizadas via environment variables

¿POR QUÉ HEXAGONAL?
- Desacoplamiento: Cambiar un adapter no afecta la lógica central
- Testeable: Cada módulo puede ser testeado aisladamente
- Extensible: Nuevos adapters (Telegram, SMS) sin modificar el core
- Multi-tenant: Config-driven permite múltiples clientes en un deployment

4. MÓDULOS Y COMPONENTES
================================================================================

El sistema utiliza 8 módulos independientes conectados por el MessageProcessor:

4.1 INTENT CLASSIFIER
Entrada: Mensaje de cliente
Salida: ClassificationResult (intent, temperatura, confianza)
Modelo: qwen2.5:7b-instruct
Intents válidos: product_inquiry, pricing_request, appointment_request, complaint,
                 support_request, follow_up_reply, unknown
Temperaturas: hot, warm, cold, unqualified

4.2 RESPONSE ORCHESTRATOR
Entrada: Clasificación + contexto del tenant
Salida: OrchestratorResult (should_send, response_text, escalated)
Modelo: qwen2.5:14b-instruct (fallback a 7b)
Contexto: business_name, brand_tone, FAQ, business_hours, knowledge_documents

4.3 CRM WRITER
- CRUD de contactos, leads, conversaciones, mensajes
- Upsert automático: no duplica registros
- Almacena raw payload y mensajes normalizados

4.4 HUMAN ESCALATION
- Lógica de decisión: ¿debe escalar?
- Triggers: complaint_intent, customer_requests_human, config rules
- IMPORTANTE: Al escalar, TAMBIÉN genera respuesta empática al cliente

4.5 KNOWLEDGE RESOLVER
- Resuelve contexto del tenant: FAQs, tone, horarios, signature
- Inyecta en el prompt del LLM para respuestas contextualizadas

4.6 FOLLOW-UP ENGINE
- Planifica jobs de seguimiento para hot/warm leads
- Persiste en PostgreSQL, ejecutado por RQ worker

4.7 ENTITY EXTRACTOR
- Extrae información del mensaje (nombre, email, teléfono, intención)
- Enriquece con datos del contacto ya conocidos

4.8 AUDIT LOGGER
- Registra TODOS los eventos (inbound, classified, escalated, responded)
- Trazabilidad completa para compliance y debugging

6. ISSUES RESOLVIDOS
================================================================================

ISSUE #1: 401 Unauthorized en /api/tenants
Causa: API_SECRET_KEY no estaba en docker-compose.yml
Solución: Añadido a environment variables + docker compose up -d --no-deps api

ISSUE #2: Environment variables no se aplicaban después de docker restart
Causa: docker compose restart NO recrea el container, solo lo reinicia
Solución: Usar docker compose up -d --no-deps api para recrear con nuevas env vars

ISSUE #3: Emojis rompiendo PowerShell script
Causa: Emojis en strings PS1 causan errores de encoding
Solución: Remover emojis del script, mantener texto plano

ISSUE #4: Meta webhook 'Verification failed'
Causa: Mismatch en WHATSAPP_VERIFY_TOKEN (config vieja debido a lru_cache)
Solución: Recrear container api con nuevas env vars

ISSUE #5: PowerShell curl syntax no compatible
Causa: PowerShell alias curl → Invoke-WebRequest con sintaxis diferente
Solución: Usar Invoke-RestMethod con hash de headers explícito

ISSUE #6: Caracteres UTF-8 garbled en terminal PowerShell
Causa: Encoding de terminal Windows, no del sistema
Solución: Cosmético, respuestas en API son UTF-8 correcto

7. PROBLEMAS ABIERTOS
================================================================================

PROBLEMA #1: Meta Sandbox Phone Number Registration
Descripción: Número de prueba en Meta sandbox no completamente registrado
Error: 'Object does not exist or missing permissions' (code 100, subcode 33)
Impacto: No se pueden enviar mensajes de prueba desde Meta a Ollama en sandbox
Causa RAÍZ: Limitación de Meta sandbox, NO un bug del código
Resolución: Usa Twilio sandbox (sin restricción) O usa número real de WhatsApp
            Business verificado

PROBLEMA #2: Latencia de Ollama (40-60 segundos por request)
Descripción: Sin GPU, Ollama tarda 40-60 segundos por clasificación + generación
Impacto: User experience pobre, cliente espera >1 minuto por respuesta
Resolución: Migrar a Together.ai (2-4 segundos, $0.20/1M tokens) OR comprar VPS
            con GPU

PROBLEMA #3: RQ Worker Status 'Unhealthy'
Descripción: docker ps muestra healthcheck UNHEALTHY en worker
Impacto: Follow-up jobs podrían no ejecutarse
Resolución: Añadir healthcheck al service worker en docker-compose.yml
